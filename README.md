## Задача 1: API Gateway 

Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.

Решение должно соответствовать следующим требованиям:
- маршрутизация запросов к нужному сервису на основе конфигурации,
- возможность проверки аутентификационной информации в запросах,
- обеспечение терминации HTTPS.

Обоснуйте свой выбор.

##Решение 1:
Вот основные кандидаты для сравнения:

| Характеристика | **Kong** | **NGINX** | **Traefik** | **HAProxy** |
|----------------|----------|-----------|-------------|-------------|
| **Маршрутизация на основе конфигурации** | ✅ Да (через admin API или декларативно) | ✅ Да (через конфиг файлы) | ✅ Да (автоматически через провайдеры) | ✅ Да (через конфиг файлы) |
| **Проверка аутентификации** | ✅ Встроенные плагины (JWT, OAuth, Key Auth) | ✅ Через auth_request модуль | ✅ Встроенный middleware (JWT, Basic Auth) | ⚠️ Требует внешних решений |
| **Терминация HTTPS** | ✅ Да | ✅ Да | ✅ Да | ✅ Да |
| **Поддержка микросервисов** | Отличная (специализирован) | Хорошая | Отличная (cloud-native) | Средняя |
| **Сложность настройки** | Средняя | Выше средней | Низкая | Средняя |
| **Масштабируемость** | Высокая | Высокая | Высокая | Высокая |
| **Динамическое обновление** | ✅ Да (без перезапуска) | ⚠️ Требует перезагрузки | ✅ Да (без перезапуска) | ⚠️ Требует перезагрузки |
| **Community/поддержка** | Большая | Огромная | Растущая | Большая |

Я рекомендую выбрать **Kong** как основное решение по следующим причинам:

1. **Специализирован под API Gateway** - в отличие от NGINX и HAProxy, которые являются веб-серверами/балансировщиками, Kong создан именно как API Gateway

2. **Богатая экосистема плагинов** - для аутентификации есть готовые плагины (JWT, OAuth2, Basic Auth), не нужно писать дополнительный код

3. **Удобное управление** - есть Admin API и GUI (Kong Manager), что упрощает эксплуатацию

4. **Масштабирование** - отлично работает в кластере, что важно для микросервисов

5. **Open Source** - есть бесплатная версия с необходимым функционалом

**Альтернативный вариант**: Если важна простота и вы уже используете Kubernetes - **Traefik** будет отличным выбором как cloud-native решение.

## Задача 2: Брокер сообщений

Составьте таблицу возможностей различных брокеров сообщений. На основе таблицы сделайте обоснованный выбор решения.

Решение должно соответствовать следующим требованиям:
- поддержка кластеризации для обеспечения надёжности,
- хранение сообщений на диске в процессе доставки,
- высокая скорость работы,
- поддержка различных форматов сообщений,
- разделение прав доступа к различным потокам сообщений,
- простота эксплуатации.

Обоснуйте свой выбор.

##Решение 2:

| Характеристика | **RabbitMQ** | **Apache Kafka** | **ActiveMQ** | **Redis** |
|----------------|--------------|-------------------|--------------|-----------|
| **Кластеризация** | ✅ Отличная | ✅ Отличная | ✅ Хорошая | ✅ Хорошая |
| **Хранение на диске** | ✅ Да (persistent queues) | ✅ Да (лог на диске) | ✅ Да | ⚠️ Опционально (Redis persistence) |
| **Скорость работы** | Высокая | Очень высокая | Средняя | Очень высокая |
| **Форматы сообщений** | Любые (JSON, XML, binary) | Любые (Avro, JSON, binary) | Любые | Любые |
| **Разделение прав доступа** | ✅ Да (vhosts, permissions) | ✅ Да (ACLs, SSL) | ✅ Да | ✅ Да (ACLs) |
| **Простота эксплуатации** | Простой | Сложный | Средний | Очень простой |
| **Протоколы** | AMQP, MQTT, STOMP | Собственный протокол | OpenWire, STOMP, MQTT | RESP |
| **Гарантии доставки** | Все уровни (at most/least once) | Exactly once | Все уровни | At most once |
| **Мониторинг** | Отличный (Prometheus, Grafana) | Хороший | Хороший | Хороший |

Я рекомендую выбрать **RabbitMQ** как оптимальное решение:

1. **Лучший баланс характеристик** - сочетает высокую производительность с надёжностью и простотой

2. **Хранение на диске** - persistent queues гарантируют, что сообщения не потеряются даже при сбоях

3. **Кластеризация** - отличная поддержка кластеров и очередей с репликацией (mirrored queues)

4. **Простота эксплуатации**:
   - Интуитивный web-интерфейс
   - Понятная конфигурация
   - Большое количество документации
   - Лёгкий запуск через Docker

5. **Гибкость маршрутизации** - различные exchange types (direct, topic, fanout, headers)

6. **Поддержка множества протоколов** - AMQP, MQTT, STOMP, что даёт гибкость при интеграции

7. **Управление доступом** - vhosts и детальные права для пользователей

**Альтернативный вариант**: Если важна максимальная производительность и обработка огромных потоков данных - стоит рассмотреть **Apache Kafka**, но он сложнее в эксплуатации.
